# Desenvolvimento ágil
Devido a constante mudanças no ambiente empresarial, as empresas precisam responder às novas oportunidades. Como essas empresas operam em um ambiente dinâmico, é praticamente
impossível derivar um conjunto completo de requisitos de software estáveis. Esses
requisitos mudam porque os clientes acham impossível prever como um sistema irá afetar as práticas profissionais, como irá interagir com outros sistemas e quais
operações de usuário devem ser automatizadas. Por isso pode acontecer de somente após a entrega, e depois que os usuários tenham adquirido experiência com
ele, que os verdadeiros requisitos fiquem claros. Para que conseguissem um desenvolvimento mais rápido do software surgiu a ideia de "métodos ágeis" como a Programação Extrema.
Esses métodos são concebidos para produzir software útil de maneira rápida. Todos eles compartilham uma série de características comuns:
 
* Os projetos e implementação são intercalados  
* O sistema é desenvolvido em uma série de incrementos  
* Podem ser utilizadas ferramentas de teste automatizado  
* Priorizando indivíduos e interações mais que processos e ferramentas  
* Software em funcionamento mais que uma documentação abrangente  
* Colaboração com o cliente mais que negociação de contratos  
* Responder a mudanças mais que seguir um plano

Eles permitiram que o time de desenvolvimento se concentrasse no próprio software em vez de no projeto (design) ou na documentação,
com objetivo de reduzir a burocracia do processo, e o excesso de documentação desnecessária.

# Desenvolvimento por plano  
Até o início dos anos 90 acreditava-se que a maneira mais indicada de obter um software melhor era por meio do planejamento cuidados do projeto, da garantia de qualidade formalizada, do uso de métodos de
análise e projeto (design) apoiados por ferramentas de software e de processos de desenvolvimento controlados e rigorosos.
Para alguns softwares, como de controle críticos em segurança, onde uma análise mais completa do sistema é essencial, a abordagem dirigida por plano é mais indicada.
Em um processo de desenvolvimento de software dirigido por plano, a iteração ocorre dentro das atividades, com documentos
formais sendo utilizados como meio de comunicação entre as etapas do processo.

# O papel de histórias de usuário na elicitação de requisitos  
As histórias de usuário substituem uma atividade formal e separada de levantamento de requisitos. Seu papel é integrar a elicitação de requisitos ao próprio processo de desenvolvimento ágil.
Os métodos ágeis não têm uma atividade de engenharia de requisitos específica ou independente. Em vez disso, a elicitação dos requisitos é integrada ao desenvolvimento.
Criando cenários baseados na experiência do usuário, foi desenvolvida a ideia de histórias do usuário, com o intuito de formar cenários de uso baseados nas experiências de um usuário do sistema.  
* Envolvem colaboração entre cliente e equipe  
O cliente do sistema trabalha em colaboração com o time de desenvolvimento e discute esses cenários com os membros do time. Juntos, eles desenvolvem um 'cartão de história' que descreve resumidamente uma história que reúna as necessidades do usuário.
* Servem ao planejamento de iterações  
As histórias do usuário podem ser utilizadas no planejamento das iterações do sistema. Depois de desenvolvidos, os cartões de história devem ser decompostos em tarefas pelo time de desenvolvimento, que estima o esforço e os recursos necessários para implementar cada uma delas. 
* São priorizadas pelo cliente  
O cliente prioriza as histórias a serem implementadas, escolhendo as que podem ser utilizadas imediatamente para proporcionar suporte útil ao negócio.
* Facilitam a adaptação a mudanças  
A medida que os requisitos mudam, as histórias não implementadas também mudam ou são descartadas. Se forem necessárias alterações em um sistema que já foi entregue, são elaborados novos cartões de história e, mais uma vez, o cliente decide se essas alterações devem ter ou não prioridade sobre novas funcionalidades.
